## 一、OpenClaw 是什么？它要解决什么问题？

### 设计哲学

OpenClaw 的核心理念是：**每个人都应该拥有一个随时在线、记得所有对话、能跨平台与你交流的 AI 助手**。

它要解决的核心问题：
- **碎片化**：你可能在 Telegram 上问了 AI 一个问题，但在 WhatsApp 上又得重新说一遍上下文——OpenClaw 帮你统一管理
- **被动性**：传统 AI 聊天只能你问它答——OpenClaw 的 AI 可以定时检查任务、主动提醒你
- **单一性**：一个 AI 人格不够用——OpenClaw 允许你创建多个独立的 AI 助手，各司其职
- **只会说不会做**：普通聊天 AI 只能给建议——OpenClaw 的 AI 能真正"动手"，读写文件、执行命令、操作浏览器

### 它不是什么

- **不是云服务**：OpenClaw 运行在用户自己的电脑或服务器上，数据完全自主掌控
- **不是单一的聊天应用**：它是一个平台，通过各种"渠道"与用户交流
- **不是一个 AI 模型**：它是调度和管理 AI 模型的基础设施，可以对接多种 AI 模型提供商

## 二、整体架构：Gateway 是一切的中枢

OpenClaw 的架构中心是一个叫 **Gateway** 的常驻服务。所有消息——无论来自 Telegram、WhatsApp 还是网页聊天——都要经过它。

```
           ┌─────────────┐
           │   用户设备    │
           └──────┬───────┘
                  │
    ┌─────────────┼─────────────┐
    │             │             │
    ▼             ▼             ▼
Telegram      WhatsApp      CoClaw App
  渠道          渠道          渠道
    │             │             │
    └─────────────┼─────────────┘
                  │
                  ▼
         ┌────────────────┐
         │    Gateway      │
         │  （中枢服务）    │
         │                │
         │  · 接收消息     │
         │  · 路由到 Agent │
         │  · 管理会话     │
         │  · 调度 AI 模型 │
         │  · 推送回复     │
         └────────────────┘
                  │
                  ▼
         ┌────────────────┐
         │   AI 模型提供商  │
         │  （Claude, GPT  │
         │   Gemini...）   │
         └────────────────┘
```

Gateway 的主要职责：
- **统一接入**：所有渠道的消息都汇聚到这里
- **会话管理**：维护每一段对话的历史记录
- **Agent 调度**：决定哪个 AI "人格"来处理这条消息
- **模型调用**：将对话发送给 AI 模型，获取回复
- **回复投递**：把 AI 的回复发回给用户所在的渠道

Gateway 通过 **WebSocket** 与客户端保持长连接，实时双向通信。连接建立后：
- Gateway 每 30 秒发送一次心跳信号（tick），客户端据此检测连接是否存活
- 如果超过 60 秒没收到心跳，视为连接中断
- 断线后客户端需要重新连接并刷新状态（Gateway 不会重放错过的事件）

## 三、渠道（Channel）：消息从哪来、回哪去

### 什么是渠道

渠道是用户与 AI 交流的"入口"。OpenClaw 通过**插件体系**支持多种渠道。每种 IM 平台（Telegram、WhatsApp、Slack、Discord 等）都是一个独立的 Channel 插件。

### 渠道的工作方式

以 Telegram 为例：

1. **监听**：Telegram 插件启动一个监听器，持续接收来自 Telegram 服务器的新消息
2. **收消息**：用户在 Telegram 上给 Bot 发了一条消息，监听器捕获到
3. **构建上下文**：插件提取关键信息——谁发的（sender）、什么类型的聊天（私聊/群组/论坛话题）、对方的标识（peerId）
4. **路由**：Gateway 根据规则决定这条消息交给哪个 Agent 处理、写入哪本"会话笔记本"
5. **安全检查**：检查发送者是否被允许与 AI 对话（私聊策略、白名单、群组策略等）
6. **AI 处理**：消息交给 Agent 执行
7. **发送回复**：AI 的回复通过 Telegram 插件的"出站"能力发回 Telegram

每种渠道有自己的独特能力：
- **Telegram**：支持论坛话题、表情回应、投票、内联按钮
- **Discord**：支持帖子线程、角色权限、服务器频道
- **Slack**：支持消息线程、工作区权限
- **WhatsApp**：支持语音消息、图片

### 渠道不只是 IM 平台

除了 IM 平台，OpenClaw 还有两种内置的"渠道"：
- **WebChat**：OpenClaw 自带的网页聊天界面，通过 WebSocket 直连 Gateway
- **CLI**：命令行界面，开发者可以直接在终端与 AI 对话

CoClaw 的定位类似于 WebChat——通过 WebSocket 与 Gateway 通信，但提供了更丰富的产品体验。

## 四、WebChat 与 IM Channel 的本质差异

这是理解 CoClaw 产品边界的关键。WebChat（以及 CoClaw）与 IM 渠道在底层机制上有根本性差异：

| 维度 | WebChat / CoClaw | IM 渠道（Telegram 等） |
|------|-----------------|---------------------|
| **本质** | Gateway 内置的 RPC 接口 | 通过插件 SDK 注册的独立 Channel 插件 |
| **通信方式** | WebSocket 直连 Gateway | 各平台独立协议（轮询/webhook） |
| **会话分配** | **调用方直接指定**会话标签 | 系统根据规则**自动计算**会话标签 |
| **消息到达** | 即时，无中间层 | 经过 IM 平台服务器 |
| **回复方式** | 实时流式推送（边生成边展示） | 分块合并后发送（攒一段发一段） |
| **附件处理** | 图片内嵌 base64 | 下载/上传到平台 API |
| **独有能力** | 中断回复、注入消息、加载历史 | 表情回应、投票、线程、群组 |

**最关键的差异是"会话分配"**：

- IM 渠道：用户在 Telegram 上发消息时，系统自动根据"谁发的"、"从哪个平台发的"等信息计算出会话标签（Session Key），用户无法控制
- WebChat / CoClaw：我们在发送消息时直接告诉 Gateway "请把这条消息放进标签为 xxx 的会话"，完全由我们掌控

这意味着 CoClaw 拥有更大的灵活性——我们可以自由实现多会话管理、会话切换等功能，不受 OpenClaw 内置路由规则的约束。

## 五、Agent：AI 的"人格"

### 顶层 Agent（Top-Level Agent）—— 独立的 AI 个体

OpenClaw 支持创建多个**顶层 Agent**，每个都是一个独立的 AI 人格。默认有一个叫 `main` 的 Agent，用户还可以通过命令行创建更多。

每个顶层 Agent 拥有**完全独立**的一切：

| 资源 | 说明 | 示例 |
|------|------|------|
| 工作间（Workspace） | Agent 工作的"项目文件夹"，包含指令文件、项目代码等 | main → `~/.openclaw/workspace/`<br>work → `~/.openclaw/workspace-work/` |
| 会话存储 | 所有对话记录 | 各自独立的 `sessions/` 目录 |
| 认证凭据 | AI 模型的 API Key、OAuth Token 等 | 各自独立的 `auth-profiles.json` |
| 性格设定 | 通过工作间中的 `SOUL.md` 文件定义 AI 的个性和行为 | 各 Agent 可以有完全不同的人设 |

**核心观点：`main` 不是"主" Agent——它只是默认创建的第一个。所有顶层 Agent 地位平等。**

### 典型使用场景

- **多人共享**：一家人共用一个 OpenClaw 服务器，每人一个 Agent，对话完全隔离
- **多种人格**：一个人创建多个 Agent——"work" 用高端模型处理复杂任务，"quick" 用轻量模型快速回复
- **按渠道分流**：WhatsApp 消息自动路由到"生活助手"，Discord 消息路由到"技术专家"

### 消息如何路由到正确的 Agent

当消息从 IM 渠道到达时，Gateway 通过**路由规则（Bindings）**决定交给哪个 Agent。匹配优先级从高到低：

1. **精确匹配**：这个特定用户的私聊 → 交给某个 Agent
2. **服务器+角色**（Discord）：这个服务器中有某角色的成员 → 交给某个 Agent
3. **团队**（Slack）：这个工作区的消息 → 交给某个 Agent
4. **渠道账号**：这个 WhatsApp 号码收到的消息 → 交给某个 Agent
5. **渠道**：所有来自 Telegram 的消息 → 交给某个 Agent
6. **兜底**：以上都不匹配 → 交给默认 Agent（通常是 `main`）

**对 CoClaw 的影响**：CoClaw 通过 WebSocket 直连 Gateway，不走这套路由规则。但理解这个机制有助于我们未来设计"多 Agent 切换"功能。

### 子 Agent（Sub-Agent）—— 临时的助手团队

与顶层 Agent 不同，子 Agent 是 AI 在处理复杂任务时**自己临时创建的助手**。用户不能直接创建子 Agent。

**工作方式**：
```
用户问了一个复杂问题
  ↓
主 Agent 判断：这个任务需要并行处理
  ↓
主 Agent 派出 3 个子 Agent，各自负责一部分
  ↓
子 Agent 们并行工作，各自独立调用工具
  ↓
子 Agent 完成后，把结果汇报给主 Agent
  ↓
主 Agent 综合所有结果，统一回复用户
```

**可以类比**：主 Agent 是项目经理，子 Agent 是临时抽调来的专项成员。项目经理分配任务、收集结果，最后统一向客户（用户）汇报。客户不需要直接和专项成员打交道。

**关键规则**：
- 子 Agent 继承父 Agent 的工作间，但不继承性格设定（`SOUL.md`）
- 最多嵌套 2 层（项目经理 → 小组长 → 组员）
- 每个 Agent 同时最多 5 个子 Agent，全局最多 8 个并发
- 子 Agent 完成后约 1 小时自动清理
- 子 Agent **不直接与用户通信**——结果只能汇报给父 Agent

**对 CoClaw 的影响**：子 Agent 的活动在工具调用事件中可见（"AI 正在创建子任务..."）。产品需要决定在 UI 中展示到什么粒度。

## 六、Session：对话的记忆

### 基本概念

Session 是 AI 的"对话记忆"。每段对话都有一个 Session，AI 靠它来记住之前聊过什么。

**比喻**：Session 就是一本"聊天笔记本"。每次你和 AI 对话，消息都会追加到这本笔记本里。AI 在回复你时，会翻阅整本笔记本来理解上下文。

### 笔记本的物理存储

每条消息以一行 JSON 的形式追加到一个 `.jsonl` 文件中（逐行追加，永不修改已有内容）。文件存储在 Agent 的 sessions 目录下。

每行记录包含：
- **角色**：是用户说的（user）、AI 回的（assistant）、还是工具返回的结果（toolResult）
- **内容**：文本、图片、思考过程、工具调用指令等
- **时间戳**：精确到毫秒

### 一个会话的生命周期

```
1. 创建 — 第一条消息到达 → 系统分配会话标签 → 生成笔记本编号 → 开始记录
2. 活跃 — 后续消息不断追加到同一本笔记本
3. 重置 — 用户发送 /new 或 /reset → 旧笔记本归档 → 换一本新笔记本 → 标签不变
4. 归档 — 旧笔记本的文件保留在磁盘上（重命名为 *.reset.<时间戳>），可以回溯查看
5. 孤立 — 因配置变更等原因，标签从索引中删除，但笔记本文件仍然存在
```

需要注意：**首个会话并非 OpenClaw 启动时预先创建的**——它是在第一条消息到达时才按需创建。此外，OpenClaw 可能因为"会话新鲜度过期"等策略自动重置会话（相当于悄悄换了一本新笔记本），且不会发出专门的通知事件。

## 七、Session Key 与 Session ID：标签与编号

这两个概念是理解 OpenClaw 会话机制的基础。

### Session Key —— 笔记本封面上的标签

Session Key 是一个字符串，标识"这段对话应该放进哪本笔记本"。它是**逻辑路由标识**——决定消息归属到哪个上下文。

Session Key 通常不会变化（除非修改了路由配置）。它的格式是有规律的：

| 场景 | Session Key 示例 | 含义 |
|------|-----------------|------|
| 主会话 | `agent:main:main` | main Agent 的默认主会话 |
| Telegram 私聊 | `agent:main:telegram:direct:123456` | main Agent 与 Telegram 用户 123456 的私聊 |
| Telegram 群组 | `agent:main:telegram:group:chat789` | main Agent 在某 Telegram 群组中的会话 |
| 论坛话题 | `agent:main:telegram:group:chat789:topic:5` | 群组中第 5 个话题的会话 |
| 子 Agent | `agent:main:subagent:a1b2c3d4` | 主 Agent 派出的一个子 Agent 的会话 |
| 定时任务 | `agent:main:cron:daily-report` | 一个定时任务的会话 |

**规律**：Session Key 的格式始终以 `agent:<agentId>:` 开头，后续部分根据来源和规则不同而不同。

### Session ID —— 笔记本的具体编号

Session ID 是一个 UUID（如 `5c8e3a2f-...`），标识具体的 transcript 文件。每次重置会话，会生成新的 Session ID。

### 两者的关系

```
标签（Session Key）                        编号（Session ID）
"agent:main:main"           ──映射到──→    5c8e3a2f-...
"agent:main:telegram:direct:123"  ──→      a1b2c3d4-...
```

- 一个标签在任意时刻只关联一个编号
- 重置会话 = 保持标签不变，但换一个新编号（新笔记本）
- 旧的笔记本文件不删除，只是不再被标签指向

### 孤立会话（Orphan Session）

当一个 Session ID 对应的笔记本文件还在磁盘上，但标签已经从索引中删除时，这个会话就变成了"孤立会话"。常见原因：
- 会话被重置后，旧的笔记本不再被标签指向
- 路由配置变更导致标签消失

**这对 CoClaw 有实际影响**：如果用户想继续一个历史会话，需要通过特殊方式（直接指定 Session ID 而非 Session Key）来续写。

## 八、dmScope：私聊消息归入哪本笔记本

### 问题背景

当用户通过 IM 平台（如 Telegram）给 AI 发私聊消息时，Gateway 需要决定：**这条消息归入哪个 Session Key？**

这个决策由一个叫 **dmScope** 的全局配置项控制。它只影响 IM 渠道的私聊消息（不影响群组消息、WebChat、CoClaw）。

### 四种策略

**场景假设**：用户张三在 Telegram 上和 AI 私聊，用户李四在 WhatsApp 上和 AI 私聊。

#### 策略一：共享主会话（main）

所有私聊消息都写进同一本笔记本 `agent:main:main`。

```
张三（Telegram）──→ agent:main:main ←── 李四（WhatsApp）
                         ↑
              同一本笔记本，AI 能看到所有人的消息
```

- 优点：简单，适合只有自己一个人使用
- 缺点：多人场景下有**隐私风险**——张三能通过 AI 的回复间接感知到李四的对话内容

#### 策略二：按人隔离（per-peer）

每个人有自己的笔记本，但同一个人**跨平台共享**。

```
张三（Telegram）──→ agent:main:direct:zhangsan
张三（WhatsApp）──→ agent:main:direct:zhangsan  ← 同一本！
李四（WhatsApp）──→ agent:main:direct:lisi       ← 独立
```

- 优点：人与人之间隔离
- 优点：同一个人换了 IM 平台，对话可以延续
- 缺点：需要识别"同一个人"——不同平台上的 ID 不同，需要借助"身份关联"功能

#### 策略三：按渠道+人隔离（per-channel-peer）

每个人在每个平台上各有一本笔记本。

```
张三（Telegram）──→ agent:main:telegram:direct:zhangsan
张三（WhatsApp）──→ agent:main:whatsapp:direct:zhangsan  ← 不同！
李四（WhatsApp）──→ agent:main:whatsapp:direct:lisi
```

- 优点：最安全，不同平台的对话完全隔离
- 缺点：同一个人换平台时，对话无法延续
- **这是绝大多数用户实际使用的模式**（安装引导流程自动设置）

#### 策略四：按账号+渠道+人隔离（per-account-channel-peer）

在策略三的基础上，进一步按 IM 账号隔离。适用于一个 IM 平台有多个账号的场景（比如两个 WhatsApp 号码）。

### 默认值辨析

这里有一个容易混淆的点：

- **代码层面的默认值**：`"main"`（共享主会话）
- **安装引导的默认值**：`"per-channel-peer"`（按渠道+人隔离）

实际效果是：用户通过正常流程安装 OpenClaw 后，dmScope 会被自动设为 `"per-channel-peer"`。只有手动编辑配置文件且删除了 dmScope 设置，才会回退到代码默认的 `"main"`。

### dmScope 是全局的

dmScope **不能按渠道或按 Agent 单独设置**。一旦设定，所有 IM 渠道的私聊消息、所有顶层 Agent 都遵循同一个策略。

### CoClaw 不受 dmScope 影响

CoClaw 作为 WebChat 类渠道，在发送消息时直接指定 Session Key，不走 dmScope 的自动计算逻辑。dmScope 只影响 IM 渠道的私聊消息。

## 九、Peer：对话的另一端是谁

### 什么是 Peer

Peer（对等方）这个概念出现在 Session Key 的构建过程中，表示"与 AI 对话的另一端"。根据聊天类型不同，Peer 的含义有所区别：

| 聊天类型 | Peer 代表 | 示例 |
|---------|----------|------|
| 私聊 | **发消息的人** | Telegram 用户 ID `123456789` |
| 群组 | **群组本身** | Telegram 群组 ID `-1001234567890` |
| 论坛话题 | **话题本身** | 群组+话题 `-1001234567890:topic:42` |

**关键点**：在群组场景中，Peer 是群组本身，不区分群里谁发的消息。这意味着群组中所有人的消息共享同一个会话上下文。

### Peer ID 的来源

每种 IM 平台有自己的用户标识体系，Peer ID 就是平台原生的标识符：

- Telegram：用户 ID（数字），如 `123456789`
- WhatsApp：手机号，如 `+8613800138000`
- Discord：雪花 ID（长数字串）
- Slack：Slack 内部频道/用户 ID

### 身份关联（Identity Links）

在"按人隔离"（per-peer）模式下，同一个人在不同平台上的 Peer ID 不同，默认会被视为不同的人。如果希望将它们关联起来，可以配置"身份关联"：

```
告诉系统：
  "alice" = Telegram 上的 123456789 = WhatsApp 上的 +8613800138000
```

这样 Alice 无论从哪个平台发消息，Session Key 中的 Peer 部分都会被替换为 `alice`，从而共享同一个会话。

**适用条件**：仅在 dmScope 不是"共享主会话"时才有意义（共享模式下所有人已经在同一个会话了）。

## 十、Thread：群组内的话题分支

### 什么是 Thread

Thread（线程/话题）是**群组场景中的消息分支功能**，不是编程中的"线程"概念。它对应各 IM 平台的原生功能：

| 平台 | Thread 形态 |
|------|------------|
| Telegram | 论坛话题（Forum Topic） |
| Discord | 帖子/频道线程 |
| Slack | 消息回复线程 |

**作用**：让同一个群组/频道内的多个话题可以分开讨论，互不干扰。

### Thread 如何影响会话

Thread 通过在 Session Key 后面追加后缀来实现隔离：

```
不带 Thread 的群组会话：
  agent:main:telegram:group:-1001234567890

带 Thread 的话题会话：
  agent:main:telegram:group:-1001234567890:topic:3
                                           ^^^^^^^^
                                        话题后缀
```

这意味着群组中的每个话题都有**独立的会话记忆**。AI 在"话题 3"中的对话不会看到"话题 5"中的历史。

### Thread Binding：线程绑定

高级功能：可以将一个 Thread 绑定到一个特定的会话（如子 Agent 会话），此后该 Thread 内的所有消息都会直接路由到绑定的会话，而不走正常的路由逻辑。

这通常用于 Discord 场景——让一个帖子线程成为与特定 AI 助手的专属对话空间。

### 对 CoClaw 的影响

Thread 是为 IM 平台的群组场景设计的。CoClaw 作为一对一的 WebChat 类产品，不需要引入 Thread 概念——用户可以通过创建多个 Session 来实现并行对话的需求，这比 Thread 更简单直观。

## 十一、消息的完整旅程：从发出到收到回复

### CoClaw 场景下的消息旅程

```
用户在 CoClaw App 中输入消息，点击发送
  ↓
CoClaw App → CoClaw Server（通过互联网）
  ↓
CoClaw Server → OpenClaw Gateway（通过内网/隧道 WebSocket）
  ↓
Gateway 收到消息：
  ├─ 查找 Session Key 对应的会话是否存在
  ├─ 如果不存在，创建新会话
  ├─ 将用户消息写入会话的 transcript 文件
  ├─ 检查是否有正在进行的 Agent Run
  │    ├─ 有 → 新消息进入排队
  │    └─ 没有 → 启动新的 Agent Run
  ↓
Agent Run 开始：
  ├─ 加载会话历史（从 transcript 文件读取）
  ├─ 构建 AI 上下文（系统提示 + 对话历史 + 工具定义）
  ├─ 调用 AI 模型
  ↓
AI 模型开始生成回复（流式推送）：
  ├─ 每生成一段文本 → 推送 assistant 事件
  ├─ 如果需要调用工具 → 推送 tool.start 事件
  │    ├─ 工具执行中 → 推送 tool.update 事件
  │    └─ 工具执行完毕 → 推送 tool.result 事件
  │    └─ AI 继续基于工具结果生成文本
  ├─ 如果需要子 Agent → 派出子 Agent 并行处理
  ↓
AI 生成完毕 → 推送 lifecycle.end 事件
  ↓
Gateway → CoClaw Server → CoClaw App（事件流实时推送）
  ↓
用户在 CoClaw App 中看到 AI 的回复逐字出现
```

### IM 渠道场景下的差异

IM 渠道的消息旅程有几个关键不同：

1. **会话标签是自动计算的**：Gateway 根据 dmScope 和发送者信息自动计算 Session Key，而不是由调用方指定
2. **回复是分块发送的**：AI 的回复不是边生成边推送，而是攒够一段（200-800 字符）后才发出一条消息，模拟人类回复节奏
3. **有安全检查**：会检查发送者是否有权与 AI 对话（私聊策略、白名单等）
4. **额外功能**：typing 指示器（"正在输入..."）、表情回应确认

## 十二、Agent Run：AI 的一次"思考任务"

### 什么是 Agent Run

每次 AI 处理用户的消息，都会启动一次 **Agent Run**。可以理解为 AI 的一次"思考-行动-回复"循环。

### Run 的生命周期事件

CoClaw 通过 WebSocket 接收 Agent Run 的实时事件流：

| 事件 | 含义 | CoClaw 应如何响应 |
|------|------|------------------|
| `lifecycle.start` | 思考开始 | 展示加载/思考中状态 |
| `thinking` | AI 的推理过程 | 可选展示（"AI 正在思考..."） |
| `assistant` | AI 生成的文本 | 实时更新回复内容 |
| `tool.start` | AI 开始调用工具 | 展示"正在执行：xxx" |
| `tool.update` | 工具执行进度 | 更新进度信息 |
| `tool.result` | 工具执行完毕 | 展示工具结果 |
| `lifecycle.end` | 思考完成 | 固化最终内容，结束加载状态 |
| `lifecycle.error` | 出错 | 展示错误信息，允许重试 |

**重要细节**：`assistant` 事件推送的文本是**完整替换模式**——每次收到的都是"目前的完整回复"，而不是增量片段。CoClaw 需要用最新收到的文本完整替换当前显示的内容。

### Run 可能包含多轮工具调用

一次 Agent Run 不只是"问一下 AI 模型"那么简单。AI 可能会：

```
第 1 轮：AI 决定先读取一个文件
  → 调用 read 工具 → 获取文件内容
第 2 轮：AI 基于文件内容，决定执行一个命令
  → 调用 exec 工具 → 获取命令输出
第 3 轮：AI 基于所有信息，生成最终回复
  → 输出文本 → Run 结束
```

每一轮都会产生对应的事件流。一次 Run 可能持续几秒到几分钟不等。

### 超时控制

- 默认 Agent Run 超时时间：600 秒（10 分钟）
- 如果 AI 在超时时间内没有完成，Run 会被强制终止

## 十三、排队与并发：一次只想一件事

### 核心规则

**同一个 Session 在同一时刻只能有一个 Agent Run 在执行**。如果 AI 还在处理上一条消息，新消息必须排队等待。

这是为了保证对话历史的一致性——如果两个 Run 并行写入同一本"笔记本"，消息顺序会混乱。

### 全局并发限制

- 不同 Session 的 Run 可以并行（默认最多 4 个）
- 子 Agent 有独立的并发限制（默认最多 8 个）

### 排队模式

当用户连续快速发消息时，队列的处理方式由配置决定：

| 模式 | 行为 | 适合场景 |
|------|------|---------|
| **收集**（默认） | 等当前 Run 结束后，把排队的多条消息合并成一轮处理 | 大多数场景 |
| **跟进** | 等当前 Run 结束后，逐条处理排队的消息 | 每条消息都需要独立回复 |
| **插入** | 把新消息注入当前正在进行的 Run | 用户想补充信息或改变方向 |
| **打断** | 终止当前 Run，用新消息启动新 Run | 用户改变了主意 |

### 防抖

用户快速连发多条文本消息时，系统默认等待 1 秒的"安静期"，将这些消息合并为一条再处理。这样可以避免"用户打了三行，AI 只看到第一行就开始回复"的情况。

媒体/附件消息和命令消息（如 `/new`）不受防抖影响，立即处理。

### 队列溢出

队列有容量上限（默认 20 条）。超出后，按策略处理多余消息（摘要化或丢弃）。

## 十四、工具系统：AI 不只是聊天

### AI 能做什么

OpenClaw 的 Agent 不仅能聊天，还能**实际操作**。内置工具包括：

| 工具类别 | 能力 | 示例 |
|---------|------|------|
| 文件操作 | 读写文件 | 阅读文档、生成报告 |
| 命令执行 | 运行 shell 命令 | 安装软件包、运行脚本 |
| 浏览器 | 打开网页、截图 | 搜索信息、填写表单 |
| 消息发送 | 向其他渠道发消息 | "把这个结果发到 Telegram" |
| 定时任务 | 创建/管理 cron 任务 | "每天早上 9 点提醒我" |
| 子 Agent | 派出子 Agent 并行处理 | 分解复杂任务 |

此外，插件可以注册自定义工具，进一步扩展 AI 的能力。

### 工具授权

不是所有工具都允许 AI 随意使用。系统支持细粒度的工具授权：

- **全局策略**：允许/禁止特定类型的工具
- **每 Agent 策略**：不同 Agent 可以有不同的工具权限。比如"work" Agent 可以执行命令，"family" Agent 不能
- **沙箱**：在隔离的容器环境中执行工具操作，防止 AI 对系统造成破坏

### CoClaw 中的工具可见性

当 AI 调用工具时，CoClaw 会收到对应的事件流（`tool.start` → `tool.update` → `tool.result`）。产品可以选择：
- **全透明**：展示工具名称、输入参数、执行结果
- **简化**：只展示"AI 正在读取文件..." / "AI 正在执行命令..."
- **折叠**：默认隐藏，用户可以展开查看

## 十五、模型与认证：AI 用谁的大脑

### 模型选择

OpenClaw 支持对接多种 AI 模型提供商（如 Anthropic Claude、OpenAI GPT、Google Gemini 等）。每个 Agent 可以配置不同的模型：

- **全局默认模型**：所有 Agent 默认使用
- **每 Agent 覆盖**：特定 Agent 使用特定模型（比如"work" Agent 用更强的模型）
- **故障转移链**：主模型不可用时，自动切换到备用模型

### 认证管理

AI 模型需要 API Key 或 OAuth Token 才能使用。OpenClaw 支持：
- 多个认证凭据同时存在（同一家提供商可以有多个 Key）
- 自动轮换：当一个 Key 达到速率限制时，自动切换到另一个
- 失败退避：Key 失败后先冷却一段时间再重试（1分钟 → 5分钟 → 25分钟 → 1小时）
- 会话粘性：同一个会话倾向于使用同一个 Key，避免频繁切换

### CoClaw 中的模型信息

CoClaw 可以展示当前会话使用的模型信息，未来也可以提供模型切换功能（对应 OpenClaw 的 `/model` 命令）。

## 十六、斜杠命令：用户的直接控制

### 什么是斜杠命令

用户可以通过以 `/` 开头的特殊消息直接控制 OpenClaw 的行为。这些命令不会作为普通消息发给 AI 处理，而是由系统直接执行。

### 关键命令

| 命令 | 功能 | 对 CoClaw 的意义 |
|------|------|-----------------|
| `/new` 或 `/reset` | 重置当前会话（换一本新笔记本） | CoClaw 可以实现"新建对话"按钮 |
| `/stop` | 中止 AI 当前的思考 | CoClaw 可以实现"停止生成"按钮 |
| `/model` | 查看/切换 AI 模型 | CoClaw 可以实现模型选择器 |
| `/status` | 查看当前上下文使用情况 | 帮助用户了解对话窗口是否快满了 |
| `/queue` | 切换消息排队模式 | 高级用户设置 |
| `/export-session` | 导出当前会话 | 会话备份/分享 |
| `/help` | 查看帮助 | 命令速查 |
| `/commands` | 查看所有可用命令 | 命令速查 |

### 命令 vs 指令

有些以 `/` 开头的内容不是独立命令，而是**指令**——它们会从消息中剥离出来，修改会话设置，剩余的文本继续作为消息发给 AI：

- `/think 请深入分析这个问题` → 开启"思考过程可见"模式 + 将"请深入分析这个问题"发给 AI
- `/model claude-opus 帮我写一段代码` → 切换到 Opus 模型 + 将"帮我写一段代码"发给 AI

### 命令授权

并非所有人都能使用所有命令。OpenClaw 支持基于白名单或频道权限的命令授权控制。

## 十七、定时任务与心跳：无人值守的自动化

### Cron 定时任务

用户（或 AI 自己）可以创建定时任务，让 AI 在指定时间自动执行操作：

- **一次性任务**：在某个时间点执行一次（如"30 分钟后提醒我"）
- **固定间隔**：每隔一段时间执行（如"每 2 小时检查一次"）
- **Cron 表达式**：灵活的时间计划（如"工作日早上 9 点"）

每个定时任务有唯一的 `jobId`，有两种执行模式：
- **Isolated 模式**（默认）：在独立会话（`agent:main:cron:<jobId>`）中执行，每次都是全新的上下文，不会污染主会话
- **Main 模式**：注入到 Agent 的主会话（`agent:main:main`）中，在心跳时执行，共享主会话的对话历史

任务完成后的结果投递需要在创建时指定：
- 仅内部执行（不通知用户）
- 向指定渠道/接收者发送结果
- 调用 Webhook

**重要特性：定时任务不区分创建来源。** 无论是哪个用户通过哪个渠道的对话触发 AI 创建了这个任务，任务一旦创建，就与来源完全脱钩——它的执行和投递由自身的配置决定，不会自动关联创建者的渠道身份。

### 心跳（Heartbeat）

心跳是一种更轻量的自动化机制：

- Gateway 定期（默认每 30 分钟）在 Agent 的**主会话**中触发一次 Agent Run
- 这里的"主会话"特指 `agent:<agentId>:main`（通常是 `agent:main:main`），与 dmScope 设置无关——即使用户的 IM 私聊被分配到了独立的 session，心跳仍然只在这个主会话中执行
- AI 检查工作间中的 `HEARTBEAT.md` 文件，判断是否有需要处理的事项
- 如果没什么事，AI 回复 `HEARTBEAT_OK`（系统会自动抑制，不投递给用户）
- 如果有事需要通知用户，AI 生成有意义的回复并投递

**心跳 vs 定时任务的区别**：
- 心跳在**主会话**（`agent:main:main`）中执行，能看到最近的对话历史
- 定时任务默认在**独立会话**（`agent:main:cron:<jobId>`）中执行，每次都是全新的上下文（也可配置为 main 模式共享主会话）
- 两者都不区分来源——心跳不属于任何特定的渠道或用户

## 十八、已知限制与边界

产品经理需要了解以下 OpenClaw 的当前限制，以便在 CoClaw 中做出正确的产品决策：

### 附件类型限制

Gateway 的 RPC 接口（CoClaw 使用的接口）目前**只支持图片附件**：
- 图片：正常处理（base64 编码后发送给 AI）
- 语音：**静默丢弃**（有日志但不报错）
- 文件：**静默丢弃**

IM 渠道则有更广泛的附件支持（因为走的是不同的处理管道）。

**产品影响**：CoClaw 如果支持语音输入，需要在客户端先转成文字再发送；如果支持文件上传，需要明确告知用户当前不支持，或等待 OpenClaw 扩展 RPC 接口。

### 非视觉模型图片丢弃

即使发了图片，如果当前 AI 模型不支持图片识别（如纯文本模型），图片也会被**静默忽略**——AI 只处理文字部分，不报错。

**产品影响**：建议在发送图片前，如果能获取到模型能力信息，在 UI 中提示"当前模型不支持图片"。

### 会话自动重置

OpenClaw 可能因为以下原因自动重置会话（换新笔记本）：
- 会话新鲜度过期（距离上次对话超过一定时间）
- 上下文窗口即将满载时的自动压缩

自动重置**不会发出专门事件通知客户端**。

**产品影响**：如果产品需要感知 Session ID 变化，需要在发送前后主动查询检测。如果产品不关心底层 Session ID 的变化（多数场景如此），可以忽略。

### 上下文窗口有限

AI 能"记住"的对话历史是有限的（取决于模型的上下文窗口大小）。当历史太长时，OpenClaw 会自动进行"压缩"——将较早的对话摘要化，腾出空间给新内容。

**产品影响**：长对话中 AI 可能会"忘记"很早之前的细节。可以在 UI 中展示上下文使用率，帮助用户了解何时需要开始新会话。

## 十九、CoClaw 的定位与关注点

### CoClaw 的本质

CoClaw 在 OpenClaw 生态中的定位是 **WebChat 的增强版**——通过 WebSocket 与 Gateway 通信，享有与 WebChat 相同的底层能力，但在此基础上提供：

- **网络隔离穿透**：解决用户设备与 OpenClaw 服务器不在同一网络的问题
- **移动端原生体验**：Android/iOS App
- **增强的会话管理**：多会话列表、历史浏览、会话搜索等
- **更丰富的交互**：工具调用可视化、思考过程展示等

### CoClaw 与各概念的关系速览

| OpenClaw 概念 | CoClaw 需要关心吗？ | 说明 |
|---|:---:|---|
| Session Key / Session ID | **是** | 会话管理的核心 |
| Agent Run 生命周期事件 | **是** | UI 实时展示的基础 |
| 工具调用事件 | **是** | 工具执行可视化 |
| 排队机制 | **是** | 用户等待体验 |
| 斜杠命令 | **是** | 对应 UI 功能按钮 |
| 附件限制 | **是** | UI 需做相应限制或提示 |
| 模型配置 | **中等** | 未来可能需要模型切换 UI |
| dmScope | **否** | CoClaw 不走 dmScope 逻辑 |
| Peer / Identity Links | **否** | IM 渠道专属概念 |
| Thread / Thread Binding | **否** | 群组场景专属，CoClaw 用多 Session 替代 |
| Agent 路由 / Bindings | **否** | IM 渠道消息路由专属 |
| 顶层 Agent 多人格 | **低（未来）** | 当前阶段绝大多数用户只有 main |
| 子 Agent | **低** | 透明展示在工具调用事件中 |
| Cron / 心跳 | **低** | 了解即可，CoClaw 当前不直接管理 |

## 附录：概念速查表

| 概念 | 一句话解释 |
|------|-----------|
| **Gateway** | OpenClaw 的中枢服务，所有消息都经过它 |
| **Channel** | 用户与 AI 交流的渠道（Telegram、WhatsApp、WebChat、CoClaw 等） |
| **Agent** | AI 人格实例。可以有多个，各自独立 |
| **顶层 Agent** | 用户手动创建的 Agent，拥有独立的工作间、认证和会话存储 |
| **子 Agent** | AI 在处理任务时自己临时派出的助手，自动清理 |
| **Session** | 一段对话的上下文（AI 的"聊天笔记本"） |
| **Session Key** | 会话的逻辑标签，决定消息归属到哪个上下文 |
| **Session ID** | 会话的物理编号（UUID），对应一个 transcript 文件 |
| **Orphan Session** | 标签已删但文件还在的历史会话 |
| **dmScope** | 控制 IM 私聊消息如何分配到会话的全局策略 |
| **Peer** | 与 AI 对话的另一端（私聊中是用户，群组中是群组） |
| **Peer ID** | Peer 的平台原生标识符（如 Telegram 用户 ID） |
| **Identity Links** | 将不同平台上的同一人关联为统一身份 |
| **Thread** | 群组/频道内的话题分支（Telegram 论坛话题、Discord 帖子线程等） |
| **Agent Run** | AI 处理一次用户消息的完整周期（思考+工具调用+回复） |
| **Bindings** | 消息路由规则，决定 IM 消息交给哪个 Agent |
| **斜杠命令** | 以 `/` 开头的控制指令（如 `/new`、`/stop`） |
| **Cron** | 定时任务，默认在独立会话中执行，不区分创建来源 |
| **心跳** | 定期自动触发的 Agent 检查，在主会话（`agent:main:main`）中执行 |
| **Transcript** | 对话的完整记录文件（.jsonl 格式） |
| **WebChat** | OpenClaw 内置的网页聊天界面，CoClaw 在架构上与其同层 |
